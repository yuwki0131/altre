# Tauri GUI アーキテクチャ詳細設計

## タスク概要
Tauri ベース GUI の全体アーキテクチャを整理し、ディレクトリ構成・データフロー・イベント処理方式を確定する。

## 目的
- ADR 0007 と QA (Q28〜Q35) の決定事項を設計図として落とし込み、実装タスクに引き継げる形にする。
- Rust バックエンドと React フロントエンド間の API/イベント仕様を定義する。
- TUI/GUI 共通コードの責務分離ポイントと抽象化方法を決める。

## 成果物
- `docs/design/tauri_gui_architecture.md`（新規）にアーキテクチャ図、プロセスフロー、API 定義を記載。
- フロントエンドとバックエンド双方で実装すべきコンポーネント一覧。
- Pull 型更新から Push 型更新へ拡張する際の検討事項メモ。

## 前提条件
- ADR 0007: Tauri ベース GUI フロントエンド方針の内容を理解していること。
- 既存 TUI 実装の API／イベントハンドラ構造を把握していること。

## 完了条件
- [x] Rust 側と React 側の責務分割が文書化されている。
- [x] コマンド API（tauri::command）とイベント通知のデータフォーマットが決定している。
- [x] ディレクトリ／ワークスペース構成案が提示され、後続実装タスクで参照できる。
- [x] Pull 型更新の手順と、将来の Push 型差分更新に向けた課題が列挙されている。

## メモ
- キーシーケンスはバックエンドで解決する方針（QA Q31）。TypeScript ↔ Rust のシリアライザ仕様を検討する。
- ファイル I/O は Rust 側で完結させるため、フロントエンドからのファイル参照は想定しない（QA Q33）。
